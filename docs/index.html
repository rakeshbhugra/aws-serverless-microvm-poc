AWS Lambda MicroVMs

2026-06-24 Rakesh Bhugra #firecracker #sandboxes #ryureflect

Lambda MicroVMs are a new serverless primitive (GA June 2026) for running untrusted / AI-generated code in strongly-isolated, stateful, suspend-resumable sandboxes — one VM per user session. This doc is the index for our learning effort: the mental model, the lifecycle, answers to our core questions (snapshots, resumability, baking our repo + docker compose in), and a numbered roadmap of follow-up docs and POCs we'll add to docs/ and pocs/.

01  Why we care — the ryureflect angle

ryureflect today gives every project its own privileged Docker-in-Docker container on a host daemon: bind-mount a workspace, run the project's docker compose stack inside, and docker exec Claude Code into it. It already hides everything behind a Runner seam (backend/app/runner/base.py) precisely so a microVM backend can drop in. Lambda MicroVMs is a near-exact match for that seam: per-session VM isolation, fast resume, lifecycle control, no servers to manage.

The open question

Our DinD model runs docker compose up (Postgres + Redis + FastAPI + Vite) inside the sandbox. The whole learning effort hangs on one thing: can a MicroVM run that same multi-process compose stack — and can we bake it into the image so it resumes pre-warmed? Section 07 answers this; POC-2 proves compose runs in a VM, POC-3 proves it snapshots warm.

02  The four nouns

Everything in MicroVMs is built from four resources. Get these straight and the rest follows.

03  The full lifecycle

Step through the end-to-end path from packaged code to a terminated VM. Click a node, use the buttons, or arrow keys.

→ → → → → →
Step 1 · Package & upload

Zip a Dockerfile + your app artifacts and aws s3 cp app.zip s3://bucket/. The Dockerfile's FROM is your container base image; the MicroVM OS base image (Amazon Linux 2023) is passed separately as --base-image-arn.

Step 2 · create-microvm-image (one-time, slow)

create-microvm-image pulls the zip, runs your Dockerfile on the AL2023 base, starts your app via CMD, waits for your /ready hook, then takes a Firecracker snapshot. State goes CREATING → CREATED. This is the expensive step you do once per code version.

Step 3 · run-microvm (per session, fast)

run-microvm --image-identifier … --ingress-network-connectors … --egress-network-connectors … --idle-policy …. Resumes from the snapshot — no boot, no app init. Returns a microvmId + a dedicated *.lambda-microvm.<region>.on.aws endpoint. State PENDING → RUNNING.

Step 4 · Connect over HTTPS

Mint a JWE token with create-microvm-auth-token (scoped to VM + ports + expiry). Hit the endpoint with X-aws-proxy-auth: <token> and X-aws-proxy-port: <N>. Supports HTTP/1.1, HTTP/2, WebSockets, gRPC, SSE — everything ryureflect's chat (SSE) and preview (HMR/WS) need.

Step 5 · Suspend (idle)

After the idle window, the VM suspends — memory + disk preserved — at a reduced cost. Triggered automatically by the --idle-policy or explicitly via suspend-microvm. This is the cost win over a resident DinD container.

Step 6 · Resume (traffic returns)

With autoResumeEnabled:true, the next request auto-resumes the VM (the endpoint 502s only if resume exhausts retries). Or call resume-microvm. Near-instant because it's a snapshot restore, not a boot. Processes pick up exactly where they were.

Step 7 · Terminate

When the session ends, terminate-microvm releases all resources and stops billing. Max lifetime is 8h regardless. The image stays around to launch fresh VMs for the next user.

04  Image build & the snapshot hooks

The build is where the "remembers running processes" magic happens. Two HTTP lifecycle hooks let your app gate the snapshot.

→ → → → →
Step 1 · Retrieve artifacts

Lambda assumes your --build-role-arn, pulls the zip from S3 (and any private ECR base image), and streams build logs to CloudWatch at /aws/lambda/microvms/<image-name>.

Step 2 · Execute Dockerfile on AL2023 base

Runs your Dockerfile instructions inside a fresh MicroVM started from the managed base image. This is where you clone the repo, install deps, pull container images, warm caches — anything baked here lands in the snapshot's disk.

Step 3 · Launch via ENTRYPOINT/CMD

Your CMD starts the app (and any background daemons). These running processes — not just files — are what get captured. A pre-started server resumes already-listening.

Step 4 · /ready hook (gate)

Lambda polls GET /aws/lambda-microvms/runtime/v1/ready on your declared port. Return 503 (immediately, don't hold) while still initializing; return 200 when warm. Timeout 1–3600s (readyTimeoutInSeconds). 200 triggers the snapshot.

Step 5 · Firecracker snapshot

Captures RAM of all processes + full rootfs + open FDs/connections. Image version → CREATED/SUCCESSFUL/ACTIVE. Every future run-microvm restores from exactly this point.

Step 6 · /validate hook (optional)

Lambda starts a fresh VM from the new snapshot and calls /validate to confirm it resumes correctly. Running mock payloads here also lets Lambda pre-fetch the snapshot regions you touch, speeding real cold resumes.

05  Snapshots & resumability — the gotchas

Resumability is the headline feature and the main footgun. Because every VM from one image shares identical initial state, anything "unique" generated at build time is cloned across all tenants.

  • Uniqueness must be deferred. IDs, secrets, session keys, entropy seeds generated during build are identical in every VM. Generate them after the VM starts — use the /run lifecycle hook to reset per-VM unique state.
  • CSPRNGs are fine if they read from /dev/urandom (default in Python 3.12+ secrets, Node crypto, Java 11+ SecureRandom). Snapshot-safe.
  • OpenSSL needs the patched build. Use AWS's base container image public.ecr.aws/lambda/microvms:al2023-minimal (ships openssl-snapsafe-libs) or you risk repeated randomness across resumes.
  • Network connections may go stale. Connections opened at build time may need re-establishing on resume; AWS SDK clients usually reconnect automatically. Validate DB/Redis connections on resume — directly relevant to our compose stack.
Maps to a known ryureflect concern

Postgres + Redis baked into the snapshot will resume with their old connection state. POC-3 must verify the FastAPI app re-establishes pool connections on resume (or warms them in a /run hook), exactly like the snapshot-compat guidance says.

06  Networking — how the browser reaches the VM

Each VM gets one HTTPS endpoint. Auth is a per-request JWE token; the target port is chosen by header.

→ → →
Step 1 · Choose ingress + egress connectors

At run-microvm: AWS-managed ALL_INGRESS for inbound, INTERNET_EGRESS for outbound. Connectors are immutable once the VM runs and are reused across many VMs. Swap egress for a VPC connector to reach RDS/ElastiCache.

Step 2 · Dedicated endpoint

Each VM is reachable at <id>.lambda-microvm.<region>.on.aws over TLS. No load balancer to provision. Default routing target is port 8080 inside the VM.

Step 3 · JWE auth token

create-microvm-auth-token --expiration-in-minutes 30 --allowed-ports '[{"port":5173}]'. Encrypted, scoped to one VM + a port set + expiry. Send as X-aws-proxy-auth. Wrong/expired token or out-of-scope port → 403.

Step 4 · Multi-port routing

Target a port via X-aws-proxy-port: N header, or for WebSockets the subprotocol lambda-microvms.port.N. This is how one VM exposes Vite (5173) and FastAPI (8000) — token must allow both ports.

07  Can we bake our repo + docker compose up in?

This is the question that started the whole thing. Short answer below, with the nuance that matters.

Repo pre-clone — Yes, cleanly

The Dockerfile build phase runs with internet access. RUN git clone … (or COPY the workspace in) + install deps lands in the snapshot's disk. Every VM resumes with the repo already present — no clone latency at session start. This directly replaces ryureflect's "copy golden template into workspace" step for the baked parts.

docker compose up inside — Yes, but it's the real POC

A MicroVM is a full VM with OS capabilities — you can grant --additional-os-capabilities '["ALL"]' (mount FS, netns, eBPF) and run a full Docker daemon + Compose inside, the same DinD pattern we use now. The cleaner path may be to flatten the compose stack into co-resident processes (Postgres, Redis, FastAPI, Vite as background daemons started by CMD + a process manager) so the snapshot captures them all pre-warmed. Which is better — nested Docker vs. flattened processes — POC-2 proves compose runs in a VM at all; POC-3 decides which to bake into the snapshot.

Pre-warmed & resumable — Yes, that's the point

Because the snapshot captures running processes, a compose stack started and warmed at build time resumes already-up. No docker compose up wait at session start. Caveat from §05: re-validate DB/Redis connections on resume and defer any unique IDs to a /run hook.

08  Command cheat-sheet

# 1. package + upload
zip app.zip Dockerfile app.js
aws s3 cp app.zip s3://<bucket>/app.zip

# 2. build the image (slow, once per code version)
aws lambda-microvms create-microvm-image \
  --name ryu-sandbox \
  --code-artifact uri=s3://<bucket>/app.zip \
  --base-image-arn arn:aws:lambda:us-east-1:aws:microvm-image:al2023-1 \
  --build-role-arn arn:aws:iam::<acct>:role/MicrovmBuildRole
aws lambda-microvms get-microvm-image --image-identifier ryu-sandbox   # wait for CREATED

# 3. run a VM (fast, per session)
aws lambda-microvms run-microvm \
  --image-identifier ryu-sandbox \
  --ingress-network-connectors arn:aws:lambda:us-east-1:aws:network-connector:aws-network-connector:ALL_INGRESS \
  --egress-network-connectors  arn:aws:lambda:us-east-1:aws:network-connector:aws-network-connector:INTERNET_EGRESS \
  --idle-policy '{"autoResumeEnabled":true,"maxIdleDurationSeconds":900,"suspendedDurationSeconds":300}'

# 4. connect
aws lambda-microvms create-microvm-auth-token --microvm-identifier <id> \
  --expiration-in-minutes 30 --allowed-ports '[{"allPorts":{}}]'
curl https://<id>.lambda-microvm.us-east-1.on.aws/ -H "X-aws-proxy-auth: <token>"

# 5. lifecycle
aws lambda-microvms suspend-microvm  --microvm-identifier <id>
aws lambda-microvms resume-microvm   --microvm-identifier <id>
aws lambda-microvms terminate-microvm --microvm-identifier <id>

09  The learning roadmap — docs & POCs to add

Each track becomes its own doc in docs/ and/or a runnable POC in pocs/. Status badges track progress.

  1. Hello-world MicroVM, end to end

    Walk the §03 lifecycle by hand in the POC account: build the Python sample image, run it, mint a token, curl it, suspend/resume/terminate. Goal: muscle memory for the CLI + confirm IAM roles. Done — full lifecycle verified; image identified by ARN, counter survived suspend/resume.

    donepocs/01-hello-world/

  2. Run docker compose up inside a MicroVM

    Before snapshotting a stack, prove a full Docker daemon + a docker compose stack runs inside a VM at all. Minimal case: Redis + a FastAPI backend with the hit counter in Redis. Done — compose serves through the endpoint; biggest learning is that nested-container DNS only works via the 127.0.0.2 stub + host networking, and buildkit can't use it. See the findings doc.

    donepocs/02-docker-compose/ · docs/02-docker-compose.html

  3. Bake a repo + compose stack into an image

    The core question (§07). Take a trimmed ryureflect house-stack, decide nested-Docker vs. flattened-processes, build an image where the stack is pre-warmed (gate the snapshot with /ready), and prove it resumes already-up. Measure resume latency.

    plannedpocs/03-baked-compose/ · docs/03-baked-compose.html

  4. Snapshot safety & the /run hook

    Prove the §05 gotchas empirically: show shared-entropy across VMs, then fix it with a /run hook; validate Postgres/Redis reconnect on resume. Deliver a snapshot-compat checklist.

    planneddocs/04-snapshot-safety.html

  5. Networking deep-dive: multi-port + WebSockets

    Expose Vite (HMR over WS) and FastAPI from one VM via port headers / subprotocols. Confirm SSE works for AG-UI chat. Map this to ryureflect's preview iframe + chat panes.

    planneddocs/05-networking.html

  6. Run Claude Code inside a MicroVM

    Replicate exec_claude: get claude -p streaming stream-json from inside the VM (over the endpoint or a control channel). The agent-loop heart of ryureflect on the new substrate.

    plannedpocs/06-claude-in-vm/ · docs/06-claude-in-vm.html

  7. MicroVMRunner — implement the seam

    Write a Runner impl (§01) backed by MicroVMs: create_workspace→image build, start→run-microvm, preview_url→endpoint+token, watch_files, stop→terminate. Drop-in behind ryureflect's existing contract.

    planneddocs/07-microvm-runner.html

  8. Cost & ops model

    Baseline/peak pricing, idle suspend math, image build cost, vs. our current host-DinD bill. When does MicroVMs actually win? Plus base-image deprecation/patching cadence.

10  Open questions to chase

  • Is there a boto3 / SDK surface yet, or CLI + raw API only? (the runner impl needs it)
  • How does the /run hook actually fire — HTTP like ready/validate, and on every resume or only first start?
  • Per-account concurrent-VM limits — enough for one VM per active ryureflect project?
  • Can we update an image's code without a full rebuild, or is every code change a new create/update + snapshot? (affects iteration speed)
  • File-watching inside a VM for the Files pane — inotify over a control port, or poll?
  • ARM64-only: do all our images (Claude CLI, Vite toolchain, Postgres) have clean arm64 builds?
