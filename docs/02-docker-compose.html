docker compose up inside a MicroVM

2026-06-24 Rakesh Bhugra proven #docker #dns #firecracker

It works — a Redis + FastAPI stack runs via docker compose up inside a Lambda MicroVM and serves through the endpoint. But getting there took a real debugging session: an opaque 502, a terminated VM, and a nested-container DNS wall. This is the record of what broke, how we saw inside without SSH, and the exact changes that made it green — so the next track (and MicroVMRunner) inherit the scar tissue.

02  The debugging journey

How we got from a blank 502 to a green stack. Click a node, use the buttons, or arrow keys.

→ → → → → →
Step 1 · Opaque 502

curl the endpoint → HTTP 502 Bad Gateway, empty body. Per AWS's table, 502 means "application not responding." Nothing was listening on :8080 — but the error told us nothing about why. First lesson: make curl print the real status, not a stack trace.

Step 2 · The VM had terminated (exit 17)

get-microvm → state: TERMINATED, reason "Container Stopped with Exit Code: 17". The entrypoint had set -e, so when compose up failed the script died, the VM's main process exited, and the whole VM was torn down — taking the evidence with it. Fix: drop set -e so a failed stack stays alive and inspectable.

Step 3 · CloudWatch had the real error

The VM streams stdout to /aws/lambda-microvms/<image>. Reading it: dockerd started fine, images pulled fine, then the backend build hit pip install → Temporary failure in name resolution. So: DNS failing inside a nested container, even though dockerd's own pulls resolved.

Step 4 · A shell, without SSH

To poke live we baked a tiny debug-exec server on :9000 — POST a command, it runs bash -lc on the VM host and returns stdout. Reached over the same authenticated ingress (the JWE token gates it). Now microvm.py shell "docker ps" runs anything inside the VM. (There's no documented native interactive shell; this also seeds the track-5 control agent.)

Step 5 · Diagnose the DNS precisely

From the shell: host /etc/resolv.conf = nameserver 127.0.0.2 (a VM-provided stub) and the host resolves fine. Containers pointed at 8.8.8.8 fail; forcing a container onto 127.0.0.2 resolves instantly. Egress itself is fine — raw TCP to 1.1.1.1:443 connects. So the rule: only the 127.0.0.2 stub answers DNS; public resolvers are blocked.

Step 6 · The wrong turns

"Fix A" set daemon.json dns to 8.8.8.8 — which backfired, forcing every container off the one resolver that works. Then docker build --network=host alone still failed: the build container had host-netns reachability but its resolv.conf still said 8.8.8.8. And even with daemon.json → 127.0.0.2, buildkit ignores it for build RUN steps. Three dead ends, each a real lesson.

Step 7 · Green

The combination that works: daemon.json → 127.0.0.2, host networking on every service (so the loopback stub is reachable), Redis over localhost, and install deps at container start (plain docker run gets the resolver; skip docker build entirely). count climbs 1 → 2 → 3, and a fresh VM from the rebuilt image serves with zero manual steps.

04  The DNS rule, stated plainly

The one sentence to remember

Inside a Lambda MicroVM, containers resolve DNS only through the VM's local stub at 127.0.0.2; public resolvers like 8.8.8.8 are blocked. And 127.0.0.2 is host loopback, so containers must use host networking to reach it.

  • Egress is not the problem. Raw TCP from a container to 1.1.1.1:443 connects fine. Only outbound DNS to public resolvers is blocked — a narrow, easy-to-misread restriction.
  • Image pulls fooled us. dockerd pulls images as a host process, so it uses 127.0.0.2 and works. That made it look like networking was fine — until a container tried to resolve.
  • buildkit is the stubborn one. Even daemon.json → 127.0.0.2 plus --network=host didn't get build RUN steps onto the stub. Run-time containers (plain docker run) do get it. So: no in-VM docker build — pre-pull/pre-build out of band, or install at start.

05  How we saw inside — no SSH

There is no documented native interactive shell for MicroVMs (the console's likely needs an agent). Two homemade tools carried the whole debug:

  • logs — tails the VM's CloudWatch stream (/aws/lambda-microvms/<image>). This is what surfaced the pip DNS error. Works even after the VM has terminated.
  • shell — a ~20-line debug-exec HTTP server baked on :9000; shell "cmd" POSTs a command, it runs on the VM host (where Docker lives), returns stdout. Auth is the MicroVM JWE token + the VM boundary. This let us run docker run, edit compose.yml, restart dockerd, and re-test live — no rebuild cycles.
Method note

The fast loop was: keep the VM alive on failure (drop set -e) → shell in → reproduce + fix interactively → only then bake the proven config into the image. Far cheaper than the build-snapshot-run cycle, which is several minutes each pass.

06  What this means for the real stack

  • Nested compose works, but host-networking scales badly. With many services (Postgres + Redis + API + Vite + Playwright) all on host networking, you hit port conflicts and lose network isolation. For the full ryureflect image this nudges toward flattened processes on a rich Ubuntu userland over deep nested compose.
  • Base image: the MicroVM OS base is AL2023-only, but the container base (your FROM) can be ubuntu. Heavy tools (Playwright, browsers) live in the userland you choose.
  • Warm resume is unfinished. No /ready hook means the snapshot is taken mid-init, so the first request on a fresh VM still waits on install + reconnect. Gating with /ready is the next track.
  • For snapshot-warm builds, don't build in-VM. Pre-pull/pre-build images (e.g. ECR) given buildkit's DNS limitation.

07  The working compose

services:
  redis:
    image: redis:7-alpine
    network_mode: host            # redis on localhost:6379
    restart: unless-stopped

  backend:
    image: python:3.12-slim       # stock image — no in-VM docker build
    network_mode: host            # reach the 127.0.0.2 stub + redis on localhost
    working_dir: /app
    volumes: [ "./backend:/app" ]
    environment: { REDIS_HOST: 127.0.0.1 }
    command: sh -c 'pip install --no-cache-dir -r requirements.txt
                    && uvicorn app:app --host 0.0.0.0 --port 8080'
    depends_on: [ redis ]
    restart: unless-stopped
